核心设计原则 · 基于 OpenClaw 真实约束
约束1：每次会话是全新实例 → 业务状态不能存在 agent 记忆里，必须落 PostgreSQL，每次会话通过 tia-db Skill 重建上下文
约束2：Skills 是唯一能力扩展机制 → 所有业务逻辑（评估/CC/报告/风险扫描）都写成 Skills，注册进 openclaw.json
约束3：Heartbeat 是唯一主动调度接口 → HEARTBEAT.md 驱动每日待办、入职回访、风险预警，不需要外部 Cron

HEARTBEAT.md · 主动行为的发动机
# TIA HEARTBEAT — 每30分钟执行 ## 每次心跳必做 1. 调用 tia-risk Skill，获取当前所有风险预警 2. 如果有 urgency=HIGH 的预警，立即通过飞书推送，附带建议动作 3. 检查今天是否已发送「今日待办」(查 memory/today.md) 如未发送且当前时间在 08:25–08:35： - 调用 tia-risk 获取今日优先事项 - 格式化成「今日猎头工作清单」推送飞书/Discord - 在 memory/today.md 写入 sent=true ## 入职回访检查 调用：SELECT * FROM candidates WHERE stage='placed' AND onboard_date IN (CURRENT_DATE-30, -60, -90) 如有结果，生成回访提醒推送给顾问 ## 合同到期检查 调用：SELECT * FROM clients WHERE contract_expires BETWEEN NOW() AND NOW()+7days 如有结果，提醒顾问续签并生成续签邮件草稿 ## 沉默检测 推荐报告发出超过3天客户无回复 → 生成跟进话术 候选人CC后超过48小时无回应 → 生成微信跟进消息
SOUL.md · Orchestrator（主调度 agent）
# TIA Orchestrator · Talent Intelligence Agent ## 身份 你是一个专业猎头业务调度中枢，代号 TIA。 你不直接生成业务内容，你的职责是： 理解意图 → 组装上下文 → 路由任务 → 汇总结果 ## 每次响应前必须 1. 识别任务类型（BD类 / 候选人类 / 数据查询类 / 风险预警类） 2. 调用 tia-db.get_position_context() 获取相关职位上下文 3. 根据任务类型选择子 agent： - BD开发/客户沟通/合同 → 路由给 Luna - 候选人评估/CC/报告/Offer → 路由给 Swan - 数据查询/漏斗分析 → 直接调用 tia-db 返回 ## 输出规范 - 所有业务结果必须调用 tia-db.db_write() 落库 - 推荐报告输出 Markdown，可直接发给客户 - 风险预警输出结构化 JSON，供 WebUI 渲染 - 禁止输出未经 DB 上下文支撑的候选人评分 ## 边界 不存储任何业务数据在记忆文件里，所有状态走 PostgreSQL 不猜测客户偏好，必须从 client_preferences 表读取

tia-db / SKILL.md · 最关键的 Skill
--- name: tia-db description: TIA业务数据库操作。提供候选人/职位/客户数据的读写， 以及职位上下文的完整组装（含客户偏好和历史拒绝模式）。 每次候选人评估前必须先调用 get_position_context。 version: 1.0.0 --- ## 可用函数 ### db_query(sql, params[]) 执行 SELECT 查询，返回 JSON 数组。 例: db_query("SELECT * FROM candidates WHERE position_id=$1", [pos_id]) ### db_write(table, operation, data{}) operation: insert | update | delete 自动处理 updated_at 时间戳，返回生成的 id。 ### get_position_context(position_id) 组装完整职位上下文供评估使用，包含： - 职位信息 + rubric 配置 - 客户硬性要求 + off_limits 公司名单 - 历史拒绝原因归类（最近5次） - 当前漏斗状态（各阶段候选人数） 返回结构化 JSON，直接注入评估 prompt。 ### stage_update(candidate_id, new_stage) 更新候选人阶段，同时： 1. 写 touch_records（类型: stage_change） 2. 触发对应 Webhook 事件 3. 如进入 placed 阶段，写入 onboard_date=TODAY

P0 · 没有这两个 Skills，系统跑不起来
tia-db P0 · 先建
PostgreSQL 读写接口。解决 OpenClaw 无状态的根本问题——把所有业务状态从 Markdown 记忆文件迁移到关系型数据库，让 agent 每次会话都能精确读取"张三现在在哪个阶段、上次客户拒了什么类型候选人"。
输入: sql/table/operation/data输出: rows[] / id / position_context{}
tia-risk P0 · 先建
每30分钟由 HEARTBEAT 触发的风险扫描器。这是让系统从"被动工具"变成"主动助手"的关键——顾问不用手动查，系统会在问题变成危机之前主动告警。
输入: 无（全量扫描）输出: alerts[{candidate, stage, urgency, action}]
P1 · 核心业务价值，建完立刻产生差异化
tia-assess P1
v0.10 黑盒评分的升级版。关键改进：每次评估前先读 client_preferences（历史拒绝模式、硬性要求、off_limits），让分数变成"针对这个客户的这个职位的分数"而不是通用打分。每个维度必须引用简历原文作为证据。
输入: resume_text, position_id输出: assessment{dims, overall, recommendation}
tia-cc P1
猎头最高频操作，每天 10–30 次。每次 CC 前30秒生成专属话术，覆盖开场/处理拒绝/收尾三段，结果自动记录到 touch_records，积累候选人反应数据。
输入: candidate_background, position_brief, stage输出: opening, objection_handlers[], closing, social_msg
tia-report P1
标准化推荐报告生成，输出可直接发给客户 HR。每条优势都引用简历原文，风险点客观呈现（不美化），薪资匹配分析基于 client_preferences 的实际预算范围。
输入: candidate_id输出: report_markdown (可直接发送)
tia-offer P1
Offer 谈判决策支持。不只是输出一段文字，而是结构化的情景分析：同时给顾问"说给客户的话"和"说给候选人的话"，以及时间节点建议。输入变量越完整，分析越精准。
输入: offer_params{}输出: {risk_level, client_script, candidate_script, timeline}
4阶段 · 从 v0.10 出发到真正跑起来
Phase 1 · 数据地基
第1–2周 · 完成后：agent 有了长期记忆
  • 在 openclaw-11 上安装 PostgreSQL + pgvector
  • 执行建表 SQL（5张表 + enum类型 + 索引）
  • 开发 tia-db Skill（P0），注册进 openclaw.json
  • 开发 tia-risk Skill（P0），在 Dashboard 手动测试
  • 把 v0.10 已有的评估结果回填到 ai_assessment 字段
  • 写 HEARTBEAT.md 的风险扫描 + 今日待办逻辑
验收：Discord 发「今天有什么风险」→ agent 读 DB 返回结构化预警
Phase 2 · 核心业务 Skills
第3–5周 · 完成后：agent 能执行猎头80%的日常操作
  • 开发 tia-assess（注入 client_preferences，替换 v0.10 黑盒评分）
  • 开发 tia-cc（CC话术生成，结果落 touch_records）
  • 开发 tia-report（标准化推荐报告，可直接发送）
  • 开发 tia-offer（Offer 谈判结构化分析）
  • 重写 Orchestrator SOUL.md（路由规则 + 上下文组装）
  • 写 Luna / Swan 的 SOUL.md 和 Skill 权限范围
验收：Discord 发「帮我准备CC张三」→ 30秒内输出三段话术 + 记录到 DB
Phase 3 · WebUI 接入
第6–8周 · 完成后：有了可用的操作界面
  • 在 v0.10 server/index.mjs 中扩展新端点：/candidates /positions /funnel /alerts
  • WebUI Dashboard 接 REST API，实时展示漏斗数据
  • WebSocket 接 agent 工作日志，实现实时推送
  • 飞书 Bot 接入：今日待办卡片 + 风险预警消息格式
  • 推荐报告 HTML 模板标准化，支持一键复制发送
验收：打开 WebUI → 看到今日3条预警 → 点击"生成CC话术" → 30秒出结果
Phase 4 · 数据飞轮
第9周起 · 持续运营 · 护城河开始建立
  • 每次客户反馈后自动 append 到 client_preferences.rejection_patterns
  • 成单数据积累后，用 preferred_sources 优化下次寻源优先级
  • pgvector 开始发挥作用：评估时自动检索相似历史成功案例
  • 薪资 benchmark 表从成单数据中自动提取，形成内部参考值
  • 用历史数据回归分析：哪些维度真正预测成单，调整 rubric 默认权重
验收：新职位诊断时，AI 自动引用「上次这个客户拒了薪资预期超预算10%的候选人」
⚡ 最关键的一步：先把 tia-db 和 tia-risk 两个 P0 Skills 跑起来，接上 HEARTBEAT.md。其他所有功能都建立在这个基础上。没有数据层，所有 agent 的"记忆"都是会话结束就消失的幻觉。
